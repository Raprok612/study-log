---

## **🧠 이론 정리**

### **1️⃣ 데이터모델링의 정의**

> 현실 세계의 데이터를 “데이터베이스가 이해할 수 있는 구조”로 추상화하는 과정  
> **데이터 의미를 설계하는 작업이지, 단순 표를 그리는 것이 아님**

| 모델 | 초점 | 설명 |
|------|------|------|
| **개념적 모델** | 업무 중심 | 엔터티, 관계 도출 |
| **논리적 모델** | 구조 중심 | 속성 정의, 정규화 |
| **물리적 모델** | 구현 중심 | 테이블, 인덱스, 파티션 |

---

### **2️⃣ 데이터모델링의 목적**
- 데이터 **중복 최소화**
- **업무 규칙 명확화** (정의되지 않은 의미 제거)
- 데이터 간 **일관성 확보**
- 시스템 변경 시 **확장성 강화**
- SQL 성능 및 데이터 품질 향상 기반 마련

---

### **3️⃣ 모델링의 3대 특징**
- **추상화 (Abstraction)** – 본질만 남기기  
- **단순화 (Simplification)** – 복잡한 구조를 단순하게  
- **명확화 (Clarity)** – 용어·구조·명세의 일관성  

---

## **모델링 시 유의사항**

1. **중복 최소화**  
   - 동일한 데이터가 여러 위치에서 관리되는 상황 방지  
   - 중복은 정합성 문제와 수정 비용 증가로 이어짐

2. **비유연성 최소화**  
   - 구조가 과도하게 특정 상황에 묶이지 않도록 설계  
   - 변경에 강한 모델이 좋은 모델

3. **비일관성 최소화**  
   - 이름 규칙·정의·관계 구조가 모델 전체에서 통일되어야 함

---

## **데이터모델링의 중요성 (사용자 노트와 통합 정리)**

- 모델링 단계에서 방향성이 잘못되면  
  **이후 논리 구조·SQL 단계 전체가 잘못됨**
- SQL 성능 문제의 상당수는 **모델링 실패에서 시작됨**
- 실무에서 요구사항 명세보다 더 중요한 것이  
  **“데이터 간 관계 정의”**  
- 데이터 중심 사고는 실전 SQLD와 실무 SQL 모두에서 가장 핵심

---

## **⚙️ 핵심 요약**

- 데이터모델링은 의미를 설계하는 작업  
- 개념→논리→물리 단계 구조가 명확해야 함  
- 모델링 시 중복·비유연성·비일관성 주의  
- 모델링 품질은 SQL 품질로 직결됨

## **💬 느낀점**

- 단순히 테이블을 설계하는 것이 아니라,  
  **데이터 의미를 구조화하는 과정이라는 사실을 명확히 이해했다.**
- 모델링의 방향성이 전체 SQL 품질을 결정한다는 점을 실감했다.
- 앞으로의 학습(엔터티·속성·식별자·정규화)의 기반이 되는 매우 중요한 시작점이었다.
